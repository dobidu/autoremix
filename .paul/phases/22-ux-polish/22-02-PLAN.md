<objective>

## Goal
Make `ScreenRender`'s Done state aware of whether the user came from the
remix flow (single-track) or the mashup flow (two-track). Swap header
text, button labels, and "new …" destination accordingly.

## Purpose
Currently after a mashup, the user lands on ScreenRender showing
"DONE" header, [▶ Original] [▶ Remix] [New Remix] buttons. Confusing:
- "Remix" played here is actually the mashup output
- "New Remix" returns to ModeParams which is single-track remix territory
- No way to play Track B
- Header gives no signal that this was a mashup

## Output
- `ctx_.render_is_mashup` bool flag (set by ScreenMashup, cleared by ScreenModeParams)
- ScreenRender Done state honors the flag:
  - Header: "MASHUP DONE" (vs "DONE")
  - Play buttons: ▶ Track A / ▶ Track B / ▶ Mashup (vs ▶ Original / ▶ Remix)
  - PreviewMode enum gains a TrackB variant
  - "New Mashup" button returns to ScreenMashup (vs "New Remix" → ModeParams)
  - Top waveform labeled "TRACK A" (vs "ORIGINAL")
  - Bottom waveform labeled "MASHUP" (vs "REMIX")
- "New File" still works in both modes
</objective>

## Open Questions (revisit during APPLY)

- **"New Mashup" navigation**: ScreenMashup or back to ScreenStemsReady?
  Default: ScreenMashup — user's intent is "tweak and regenerate", not
  "start a fresh mashup with new files". Going to StemsReady would lose
  template + gain context.
- **Track B button position**: between Track A and Mashup, or as the third
  in a row? Default: middle [▶ Track A] [▶ Track B] [▶ Mashup] — chronological
  (B is the second source, mashup is the result).
- **Should the Track B waveform be drawn somewhere?** Skipping for v1 — the
  button alone is enough. Future polish phase could add a 3-row waveform
  display, but the screen already feels busy.
