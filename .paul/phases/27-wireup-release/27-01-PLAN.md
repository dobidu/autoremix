<objective>

## Goal
Wire every screen call site to the native modules built in Phases 23–26
and stop calling `AudioBridge` from any screen flow. `AudioBridge.cpp/h`
+ `sidecar/` stay on disk for now — Plan 27-02 removes them after this
plan proves the native path runs end-to-end.

## Purpose
This is the v4 transition. After 27-01, the plugin produces remix +
mashup output entirely in C++ with no HTTP IPC. Loading a track,
choosing a separator (algorithmic / demucs), separating, remixing,
mashing — all native. First-launch htdemucs download is wired with a
visible progress UI inside `ScreenSeparating`.

## Output
1. `ScreenContext` extended with native callbacks:
     - `analyze_file(path) → FileAnalysis`
     - `run_separation(StemPaths in_paths, sep_id, on_complete)`
     - `run_remix(StemPaths, RemixParams, out_path, on_complete)`
   (Mashup's `run_mashup` callback already exists; reuse.)
2. Editor populates these at startup with closures that route to
   `dsp::NativeAnalysis` / `dsp::separators::*` / `dsp::engines::*`.
3. Native preset + separator registries populated from
   `dsp::presets::load_remix_presets()`, `dsp::presets::load_mashup_presets()`,
   and a static `{algorithmic, demucs}` list. No sidecar HTTP call at startup.
4. Demucs path triggers `dsp::models::ensure_htdemucs()` with a progress
   callback feeding a status string in `ScreenSeparating`. First launch:
   ~352 MB download. Subsequent launches: cached fast-path returns immediately.
5. `SidecarHealthDot` renamed to `ModelStatusDot` — green when the
   htdemucs model is cached, amber while downloading, red on download
   failure. No sidecar polling.
6. `AutoRemixProcessor` no longer calls `bridge_.startSidecar` /
   `stopSidecar`. (AudioBridge instance still exists as a member —
   27-02 removes it.)

## Open Questions (revisit during APPLY)

- **Engine string mapping for `params.preset_id`**: v3 RemixParams
  carries a `preset_id` string. Native `RemixParams` does not have
  this field — engine is resolved by looking up the preset via
  `ctx_.presets`. May need a side-table `unordered_map<string,
  NativeRemixPreset>` populated at startup for O(1) lookup. Add only
  if the linear search shows up in profiling.

- **Track length vs window size**: tracks shorter than 7.8 s (the
  htdemucs window) will be padded by the separator (already handled
  in `pack_chunk`). Verify a < 8 s clip still produces valid stems
  (no garbage from the padded region bleeding into the result).
  Smoke test during the checkpoint.

- **Sample-rate mismatch**: most DAWs run at 48 kHz. If the WAV is
  48 kHz, ScreenEmpty's analyze step will load it at 48 kHz, the
  demucs separator's 44.1 kHz guard will reject it, and the user
  sees an error. Decision: either add a transparent resample in
  `run_separation` (preferred) or surface a clear "44.1 kHz only"
  status message. Pick during APPLY based on how messy the resample
  helper gets.

- **savePreset UI**: ScreenModeParams's "Save" button POSTed to the
  sidecar in v3. After 27-01 it becomes a no-op. Either:
    a) hide the button entirely
    b) make it write a JSON to the user dir directly (mirrors v3
       behavior post-sidecar)
  Defer to APPLY; option (a) is faster, option (b) preserves feature
  parity. Note in SUMMARY either way.

- **ModelStatusDot semantics**: green / amber / red based on cache
  presence + download state is fine. But what does the dot show when
  the user has not yet picked demucs (i.e. the model is irrelevant)?
  Default to "grey + tooltip 'pick Demucs to download model'", or
  hide entirely until first demucs selection. Decide during APPLY.
