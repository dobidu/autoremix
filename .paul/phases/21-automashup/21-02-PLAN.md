# Phase 21-02 — AutoMashup C++ Glue

## Objective

Plumb the sidecar mashup capability (delivered in 21-01) through the C++
plugin layer so a future `ScreenMashup` (21-03) can call it. This sub-plan
adds types, AudioBridge method, ScreenContext fields, and editor wiring.
**No UI screen and no user-visible entry point** — that is 21-03 and 21-04.

## Acceptance Criteria

- [ ] `MashupParams` struct in `PluginTypes.h` with: `file_a`, `file_b`, `separator_id`, `stem_sources` (4 entries: vocals/drums/bass/other → "a"|"b"), `target_bpm` (with `has_target_bpm` flag), `target_key` (empty string = unset)
- [ ] `MashupResult` struct in `PluginTypes.h` with: `success`, `error_message`, `output_path`, `target_bpm`, `target_key`, `length_sec`
- [ ] `AudioBridge::mashup()` method declared in `AudioBridge.h`, implemented in `AudioBridge.cpp`
  - Builds JSON body matching `MashupRequest` schema
  - Omits `target_bpm` / `target_key` from body when unset (so sidecar anchors to A)
  - POST to `/api/v1/mashup` with `cpr::Timeout{600000}` (10 min — two separations + ML)
  - Parses response into `MashupResult`; on error returns `{success=false, error_message=...}`
- [ ] `ScreenContext` adds: `file_path_b`, `detected_bpm_b`, `detected_key_b`, `stems_b`, `mashup_target_bpm`, `mashup_target_key`, `mashup_sources` (4-key map), `mashup_output_path`
- [ ] `ScreenContext` adds a `run_mashup` callback:
  `std::function<void(MashupParams, std::function<void(MashupResult)>)> run_mashup;`
- [ ] Editor wires `ctx_.run_mashup` to spawn a detached `std::thread` calling `AudioBridge::mashup()`, bouncing the result back via `juce::MessageManager::callAsync`
- [ ] Build: `cmake --build build --parallel` succeeds — all 4 targets link (VST3, AU, Standalone, shared-code)
- [ ] No regressions: existing 5-screen flow still navigates and renders correctly (Empty→Separating→StemsReady→ModeParams→Render)

## Tasks

### T1 — Types in `PluginTypes.h`
- Add `MashupParams` struct after `RemixParams`:
  ```cpp
  struct MashupParams {
      std::filesystem::path file_a;
      std::filesystem::path file_b;
      std::string separator_id = "algorithmic";
      std::unordered_map<std::string, std::string> stem_sources;  // "vocals" → "a"|"b"
      float target_bpm     = 0.0f;     // valid only when has_target_bpm
      bool  has_target_bpm = false;
      std::string target_key;          // empty = anchor to A
  };
  ```
- Add `MashupResult` struct after `ProcessResult`:
  ```cpp
  struct MashupResult {
      bool          success = false;
      std::string   error_message;
      std::filesystem::path output_path;
      float         target_bpm  = 0.0f;
      std::string   target_key;
      float         length_sec  = 0.0f;
  };
  ```

### T2 — `AudioBridge::mashup()`
- Declare in `AudioBridge.h` after `applyRemix`:
  ```cpp
  MashupResult mashup(const MashupParams& params);
  ```
- Implement in `AudioBridge.cpp`:
  - Build `nlohmann::json` with `file_a`, `file_b`, `separator_id`, `stem_sources`
  - Conditionally add `target_bpm` if `params.has_target_bpm`
  - Conditionally add `target_key` if `!params.target_key.empty()`
  - `cpr::Post` to `/api/v1/mashup` with `Timeout{600000}`
  - On non-200 status: return `{false, "HTTP " + std::to_string(r.status_code)}`
  - Parse response; if `success == false`: return `{false, resp.value("error", "mashup failed")}`
  - On success: populate `MashupResult` from response fields (`output_path`, `target_bpm`, `target_key`, `length_sec`)
  - try/catch wrapping returns `{false, e.what()}`

### T3 — `ScreenContext` fields + callback
- Add to `ScreenContext.h` after stems block:
  ```cpp
  // ── Track B (mashup mode) ─────────────────────────────────────────────────
  juce::String         file_path_b;
  float                detected_bpm_b = 120.0f;
  juce::String         detected_key_b;
  autoremix::StemPaths stems_b;
  juce::String         mashup_output_path;
  float                mashup_target_bpm = 0.0f;
  juce::String         mashup_target_key;
  std::unordered_map<std::string, std::string> mashup_sources{
      {"vocals", "a"}, {"drums", "b"}, {"bass", "b"}, {"other", "b"}
  };
  ```
- Add callback:
  ```cpp
  std::function<void(autoremix::MashupParams,
                     std::function<void(autoremix::MashupResult)>)> run_mashup;
  ```

### T4 — Editor wiring
- In `AutoRemixEditor.cpp` constructor (where other callbacks are wired):
  ```cpp
  ctx_.run_mashup = [this](autoremix::MashupParams p,
                           std::function<void(autoremix::MashupResult)> on_complete) {
      std::thread([this, p = std::move(p),
                   on_complete = std::move(on_complete)]() mutable {
          auto result = audioProcessor.getBridge().mashup(p);
          juce::MessageManager::callAsync(
              [on_complete = std::move(on_complete),
               result = std::move(result)]() mutable {
                  on_complete(std::move(result));
              });
      }).detach();
  };
  ```

### T5 — Build verification
- Run: `cmake --build build --parallel`
- Confirm: 4 targets link (Shared-code, VST3, AU on macOS only, Standalone)
- Confirm: no new compiler warnings introduced

## Open Questions (revisit during APPLY)

- Timeout: 10 min enough? Mashup runs two separations sequentially; demucs alone is ~5 min/track on CPU. Could bump to 900s (15 min) if user complains, but oversize timeouts hide hangs. **Proposal: keep 600s; if real-world tests show demucs × 2 exceeds it, bump.**
- `mashup_sources` default: vocals=a, drums/bass/other=b (classic "vocal over instrumental" mashup). Confirm or invert.
- Should we expose source_bpm_a/b and semitone_shift_b to UI? Sidecar returns them — for now drop them from `MashupResult` to keep the C++ struct small; revisit when 21-03 designs the screen.
