<objective>
## Goal
Add a Cancel button that aborts an in-progress remix operation, and a background
timer that polls sidecar health every 5 seconds so the health dot reflects live state.

## Purpose
- Cancel: Demucs separation takes 30–120s. User needs an escape hatch.
- Health poll: The health dot currently only checks at plugin startup. If the sidecar
  crashes or starts late, the dot stays wrong until the user restarts.

## Output
- Cancel button (visible only during remix, replaces or sits next to Play)
- Pressing Cancel sets a flag checked by the background thread; status → "Cancelled."
- `juce::Timer` subclass polls `/api/v1/health` every 5s; health dot updates on UI thread
- Health poll timer starts at editor construction, stops at destruction
</objective>

<task type="auto">
  <name>Task 2: Health auto-poll timer</name>
  <files>plugin/include/PluginEditor.h, plugin/src/AutoRemixEditor.cpp</files>
  <action>
    In PluginEditor.h, add a health poll timer member after `elapsed_timer_`:
    ```cpp
    struct HealthPollTimer : juce::Timer {
        AutoRemixAudioProcessorEditor& ed;
        explicit HealthPollTimer(AutoRemixAudioProcessorEditor& e) : ed(e) {}
        void timerCallback() override {
            bool alive = ed.audioProcessor.getBridge().isServerAlive();
            juce::MessageManager::callAsync([&ed = ed, alive] {
                ed.health_dot_.setAlive(alive);
            });
        }
    };
    std::unique_ptr<HealthPollTimer> health_poll_timer_;
    ```

    In SidecarHealthDot.h, add a `setAlive(bool)` method so the timer can push state:
    ```cpp
    void setAlive(bool alive) { alive_ = alive; repaint(); }
    ```
    And add `bool alive_ = false;` as a member (remove the lambda evaluation from paint
    — evaluate on set instead). Keep the existing lambda ctor for backward compat but
    mark the dot as needing a repaint when polled.

    Actually simpler: SidecarHealthDot currently evaluates the lambda on every paint().
    Don't change its internal logic — just call `health_dot_.repaint()` from the timer
    callback (on message thread via callAsync). The dot will re-evaluate its lambda
    (which calls `isServerAlive()` synchronously — 500ms timeout). This is acceptable
    since the health poll is every 5s and the paint is fast.

    Revised approach (no changes to SidecarHealthDot.h):
    ```cpp
    struct HealthPollTimer : juce::Timer {
        AutoRemixAudioProcessorEditor& ed;
        explicit HealthPollTimer(AutoRemixAudioProcessorEditor& e) : ed(e) {}
        void timerCallback() override {
            // isServerAlive() is fast (500ms timeout); run on poll thread,
            // then repaint dot on UI thread
            bool alive = ed.audioProcessor.getBridge().isServerAlive();
            juce::MessageManager::callAsync([&ed = ed, alive]() {
                (void)alive;  // dot re-evaluates its own lambda on repaint
                ed.health_dot_.repaint();
            });
        }
    };
    std::unique_ptr<HealthPollTimer> health_poll_timer_;
    ```

    Wait — `isServerAlive()` blocks for up to 500ms on a background JUCE timer thread.
    JUCE Timer callbacks run on the message thread. So calling isServerAlive() from
    timerCallback() would block the UI for 500ms every 5s. Not acceptable.

    Correct approach: dispatch to std::thread, then callAsync:
    ```cpp
    void timerCallback() override {
        std::thread([&ed = ed]() {
            bool alive = ed.audioProcessor.getBridge().isServerAlive();
            juce::MessageManager::callAsync([&ed, alive]() {
                (void)alive;
                ed.health_dot_.repaint();
            });
        }).detach();
    }
    ```

    In constructor (after drawAndConfigComponents()):
    ```cpp
    health_poll_timer_ = std::make_unique<HealthPollTimer>(*this);
    health_poll_timer_->startTimer(5000);  // poll every 5s
    ```

    In destructor:
    ```cpp
    health_poll_timer_.reset();  // stops timer via destructor
    ```

    HealthPollTimer must be declared in PluginEditor.h as a nested struct or forward-
    declared. Since it captures `ed` by reference, the editor must outlive any pending
    callAsync. This is safe because the timer is stopped in the destructor before
    the editor's members are destroyed.
