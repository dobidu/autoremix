<objective>
## Goal
Add a `GET /api/v1/analyze` sidecar endpoint and a matching `AudioBridge::analyzeFile()`
C++ method that returns detected BPM, dominant key, and duration for a loaded audio file.

## Purpose
The UI redesign in 15-02 needs source BPM and key before it can show target-BPM and
beat-count controls. This plan builds the analysis pipeline end-to-end so 15-02 only
has to wire it into the UI.

## Output
- Python: new endpoint `/api/v1/analyze?path=<absolute-path>` → `{bpm, key, duration_sec}`
- C++: `FileAnalysis` struct in PluginTypes.h; `AudioBridge::analyzeFile(path)` method
</objective>

## AC-1: Analyze endpoint returns BPM and key
```gherkin
Given the sidecar is running and a valid WAV/MP3 path is known
When GET /api/v1/analyze?path=<path> is called
Then response JSON has {bpm: float > 0, key: string, duration_sec: float > 0}
     and bpm is within ±10% of the file's actual tempo
```

## AC-2: AudioBridge::analyzeFile returns populated struct
```gherkin
Given the sidecar is alive and a file has been loaded
When analyzeFile(file_path) is called from C++
Then it returns FileAnalysis{bpm>0, key non-empty, duration_sec>0}
     and returns FileAnalysis{} (zeroed) if sidecar unreachable
```

## AC-3: Key detection returns a note name
```gherkin
Given any audio file
When analyze is called
Then key field contains a note name from {C,C#,D,D#,E,F,F#,G,G#,A,A#,B}
     optionally suffixed with m for minor (e.g. "Am", "F#")
```
