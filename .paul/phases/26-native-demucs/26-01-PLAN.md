<objective>

## Goal
Produce a working offline tool that exports the `htdemucs` model from the
official `demucs` PyPI package to ONNX, validates the result, and records
the file size + opset version. Path-out is a tracked `tools/` directory
that runs ONCE at build/release time — it is NOT part of the plugin
runtime.

## Purpose
Phase 26-02 needs a known-good `models/htdemucs.onnx` file to wire into
ONNX Runtime + the native `DemucsSeparator`. Without this artifact (and
confidence that the export pipeline works), 26-02 cannot start.

## Output
- `tools/export_demucs_onnx.py` — Python script. Loads `htdemucs` via
  `demucs.pretrained.get_model`, runs `torch.onnx.export` with a
  representative input window, runs `onnx.checker.check_model`, and
  (if onnxruntime is installed) a smoke inference pass on a small
  test waveform to confirm the graph is runnable end-to-end.
- `tools/requirements.txt` — pinned deps (`demucs`, `torch`,
  `torchaudio`, `onnx`, `onnxruntime`, `soundfile`, `numpy`).
  Explicitly NOT used at plugin runtime.
- `tools/README.md` — one-page instructions: create venv, install,
  run the script, expected output. Document any model surgery or
  opset choices.
- `models/htdemucs.onnx` — the actual exported file. Path-tracked
  but git-ignored (too large for the repo). Generation is the
  deliverable, not the committed bytes.
- `models/.gitignore` — `*.onnx`, `*.pt`, `!.gitignore`.

The bundling decision (embed in plugin via `juce_add_binary_data` vs
download-on-demand on first launch) is intentionally DEFERRED to a
checkpoint inside this plan, once we know the actual file size.

## Open Questions (revisit during APPLY)

- **htdemucs export feasibility**: Hybrid Transformer Demucs uses a
  spectrogram branch + a time-domain branch + cross-attention. Naive
  `torch.onnx.export` may fail on the STFT layer or the attention. If
  it does, options are:
    a) Patch demucs locally to use ONNX-friendly STFT.
    b) Switch model to `mdx_extra_q` or `mdx` (4-source, simpler
       architecture, known to export cleanly).
    c) Drop ONNX entirely and use LibTorch (TorchScript .pt) — bigger
       runtime dep but fewer export surprises.
  This plan deliberately produces a clear failure mode rather than
  guessing; the user picks the path forward in 26-02 if Task 2's
  checkpoint surfaces a blocker.

- **Window size**: htdemucs default segment is ~7.8 s at 44.1 kHz.
  Using dynamic axes for the `samples` dimension is the right call —
  the C++ runtime in 26-02 will feed arbitrary-length tracks chunked
  by this window. If dynamic axes fail to export, fall back to a
  fixed window and chunk on the C++ side.

- **Bundling vs download**: see checkpoint:decision. Either path is
  technically fine; this is a UX / distribution call.

- **Model size budget**: htdemucs ONNX is ~250–350 MB. If the size
  forces "download" + user resists having a network call on first
  launch, we can revisit with a quantized variant (`htdemucs_ft_q`
  if available, or post-export INT8 quantization via
  `onnxruntime.quantization.quantize_dynamic`). Out of scope for 26-01.
