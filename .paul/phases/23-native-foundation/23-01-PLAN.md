Pull RubberBand into the build via CMake FetchContent + ship a minimal C++
wrapper `TimePitchStretcher` that replaces Python's

<task type="auto">
  <name>Task 2: Link RubberBand into SharedCode + create TimePitchStretcher.h wrapper</name>
  <files>plugin/CMakeLists.txt, plugin/include/dsp/TimePitchStretcher.h</files>
  <action>
    **plugin/CMakeLists.txt**: in the existing target_link_libraries call
    for the Plugin/SharedCode target (where cpr + nlohmann_json are
    linked), add `rubberband` to the link list. Keep PRIVATE/PUBLIC
    visibility consistent with existing entries.

    **plugin/include/dsp/TimePitchStretcher.h**: new header. Header-only
    to match the project's style. Namespace `autoremix::dsp`.

    Public API:

      class TimePitchStretcher {
      public:
          TimePitchStretcher(int sampleRate, int channels);

          // ratio < 1.0 = output shorter; ratio > 1.0 = output longer
          // (i.e. 0.7 means 70% of original duration, faster playback)
          // This matches RubberBand's "time ratio" semantics.
          void setTimeRatio(double ratio);

          // 1.0 = no shift; 2.0 = +octave; 0.5 = -octave.
          // For semitones: setPitchScale(std::pow(2.0, semitones / 12.0))
          void setPitchScale(double scale);

          // Convenience: shift by semitones (positive = up, negative = down)
          void setPitchSemitones(double semitones);

          // Process all-at-once: feeds input through RubberBand, returns
          // stretched + shifted output buffer. Use for offline rendering.
          // For real-time use, expose a streaming API in a future iteration.
          juce::AudioBuffer<float> process(const juce::AudioBuffer<float>& input);

      private:
          int sampleRate_;
          int channels_;
          std::unique_ptr<RubberBand::RubberBandStretcher> stretcher_;
      };

    Implementation notes (inside the header, all inline):
    - Construct RubberBandStretcher with sampleRate, channels,
      OptionEngineFiner | OptionProcessOffline (good default for offline)
    - process() flow:
      1. setTimeRatio + setPitchScale (already set by setters)
      2. stretcher_->setExpectedInputDuration(input.getNumSamples())
      3. stretcher_->study(...) — feed input in chunks (e.g. 1024 frames)
      4. stretcher_->process(...) — feed same input again, pull output
      5. Concatenate output chunks into a juce::AudioBuffer<float>
      6. Return

    Include guards / pragma once.
    Include juce_audio_basics + <rubberband/RubberBandStretcher.h>
    + <memory> + <vector>.

    Avoid: real-time engine options for offline use. Avoid threadsafety
    claims — wrapper is single-thread only.
  </action>
  <verify>cmake --build build --parallel — green. grep -c "TimePitchStretcher" plugin/include/dsp/TimePitchStretcher.h returns ≥3</verify>
  <done>AC-2 satisfied — wrapper compiles cleanly; AC-3 (no regression) — existing build untouched</done>

## Open Questions (revisit during APPLY)

- **RubberBand version pin**: v3.3.0 is the current stable. If FetchContent
  fails (e.g., upstream tag missing), fall back to v3.1.x or use a git
  submodule pointing at a specific commit.
- **Engine option**: Finer vs Faster. Finer is the new R3 engine (higher
  quality, more CPU). Default: Finer. Could expose as a runtime knob
  in a future polish phase if rendering takes too long on slow CPUs.
- **Offline vs streaming API**: starting offline-only (process whole
  buffer). 24/25 engines need offline; a streaming variant for real-time
  preview can come later if requested.
