<objective>

## Goal
Port the three simple remix engines (Chopped & Screwed, Slowed Reverb,
Drum and Bass) from Python to native C++. All three live in one
header-only module that consumes Phase 23's primitives
(TimePitchStretcher, NativeAnalysis, NativeStems).

## Purpose
Engines are the core "remix" feature. Once these compile, the v4 build
can produce the same three outputs the v3 sidecar does without ever
touching Python. Unlocks Phase 24-02 (EffectChainEngine) and Phase 27
wire-up.

## Output
- `plugin/include/dsp/NativeRemixEngines.h` — header-only, namespace
  `autoremix::dsp::engines`. Exposes:
    * `struct RemixParams` — same field set as v3 PluginTypes::RemixParams
      (tempo_factor, pitch_shift_semi, reverb_mix, chop_interval_ms,
      bass_boost_db, drums_tempo_factor)
    * `inline juce::AudioBuffer<float>
       chopped_and_screwed(const separators::NativeStems&, double sr, const RemixParams&)`
    * `inline juce::AudioBuffer<float>
       slowed_reverb(const separators::NativeStems&, double sr, const RemixParams&)`
    * `inline juce::AudioBuffer<float>
       drum_and_bass(const separators::NativeStems&, double sr, const RemixParams&)`
- Smoke include from `AutoRemixProcessor.cpp` so the header compiles in
  a real TU. No call-sites yet (Phase 27 wire-up).
</objective>

## AC-2: Chopped & Screwed processing chain matches Python order

```gherkin
Given a NativeStems with non-silent vocals/drums/bass/other
When chopped_and_screwed(stems, sr, params) runs with default params (0.7×, -4 semi, 0.10 reverb, 500ms chop)
Then the chain is, in order:
  1. mix 4 stems into one buffer (equal weights, min length)
  2. detect_bpm on the mix (NativeAnalysis)
  3. TimePitchStretcher with timeRatio = 1/0.7 (slow down — RubberBand semantics: ratio>1 = output longer)
  4. TimePitchStretcher pitch shift -4 semitones
  5. If pitch_shift_semi < -1, apply juce::dsp HighPass at (60 + |semi| * 4) Hz
  6. If chop_interval_ms > 0, beat-aligned stutter chop (every 4th chunk repeats)
  7. juce::dsp::Reverb with mix=0.10, room_size=0.55
  8. normalize_lufs to -14 LUFS
  And the returned buffer is non-empty
```

## AC-3: Slowed Reverb processing matches Python

```gherkin
Given NativeStems with audio
When slowed_reverb(stems, sr, params) runs with defaults (0.80×, -2 semi, 0.40 reverb)
Then the chain is:
  1. mix 4 stems
  2. time stretch by tempo_factor
  3. pitch shift by pitch_shift_semi
  4. high-shelf boost: gain_db = max(0, (1 - tempo_factor) * 8) at 5 kHz
  5. juce::dsp::Reverb room_size=0.75, wet=reverb_mix, dry=1-mix, damping=0.5
  6. normalize_lufs to -14 LUFS
  And the returned buffer is non-empty
```

## AC-4: Drum and Bass processing matches Python

```gherkin
Given NativeStems with separate vocals/drums/bass/other
When drum_and_bass(stems, sr, params) runs with defaults (1.0× vocal/other, +2 semi, +6 dB bass, 1.0 drum)
Then the chain is:
  1. detect_bpm on the drums stem (not the mix — uses the cleanest signal)
  2. auto_factor = 170 / source_bpm; drum_factor = auto_factor * drums_tempo_factor
  3. vocal_factor = drum_factor * (tempo_factor if tempo_factor != 1.0 else 1.0)
  4. time stretch drums by drum_factor
  5. bass: low-shelf boost +6 dB (split LP/HP at 250 Hz, gain LP, sum)
  6. vocals: time stretch by vocal_factor + pitch shift +2 semitones
  7. other: time stretch by vocal_factor only
  8. weighted mix: drums*1.2 + bass*1.0 + vocals*0.7 + other*0.5, /3.4
  9. normalize_lufs to -14 LUFS
  And the returned buffer is non-empty
```

## Open Questions (revisit during APPLY)

- **RubberBand "ratio>1 = longer" vs Python "factor<1 = slower"**: v3
  Python passes 0.7 to mean "70% of original duration" = faster (??)
  — but in C&S it's used as "slow down". Actually librosa's
  `effects.time_stretch(rate=0.7)` produces output LONGER than input
  (rate<1 = slower = longer). RubberBand semantics: `setTimeRatio(1.4)`
  also means longer output. So for v4 engines we invert:
  `setTimeRatio(1.0 / tempo_factor)` to get the same "slow down by
  30%" behavior. Documented in the action.

- **Drum and Bass auto_factor clamp**: if source_bpm is very low (e.g.
  60), auto_factor could exceed RubberBand's 2.0 clamp. TimePitchStretcher
  already clamps internally — the resulting drum_factor will be capped
  at 2.0, which produces "as fast as we can get". Acceptable for v1.
